Tasks

Pure business logic. No I/O, no retries, no protocol details. Just transformations, validations, and decisions.

The Philosophy of "Thin" Tasks

In Ergon, a Task is where your business logic lives. But unlike traditional frameworks, Ergon tasks are deliberately "thin." They contain only business logic‚Äîtransformations, validations, enrichments, and decisions.

They do NOT contain:

  • I/O operations (no HTTP calls, database queries, file reads)
  • Retry logic (no while loops with exponential backoff)
  • Connection management (no connection pools, session handling)
  • Protocol details (no message acknowledgments, commit offsets)

üí° Why This Matters

This constraint provides enormous benefits: your business logic can be unit tested with simple function calls. No mocks needed. The same task can work with different transports‚Äîswitch from RabbitMQ to Kafka without changing business logic.

Consumer Task Example

Implement process_transaction() and the optional lifecycle hooks:

from ergon_framework.task.mixins import AsyncConsumerTask
from ergon_framework.connector import Transaction

class OrderProcessorTask(AsyncConsumerTask):
    """Consumes orders and processes them."""

    async def process_transaction(self, transaction: Transaction):
        # Parse and validate
        order = OrderSchema(**transaction.payload)
        
        # Use an injected service for enrichment
        customer = await self.crm_service.get_customer(order.customer_id)
        
        # Pure business logic
        total = sum(item.quantity * item.price for item in order.items)
        
        return {
            "order_id": order.id,
            "customer_name": customer.name,
            "total": total,
        }

    async def handle_process_success(self, transaction, result):
        self.logger.info(f"Processed order {result['order_id']}")

    async def handle_process_exception(self, transaction, exception):
        self.logger.error(f"Failed: {transaction.id}", exc_info=exception)

Dependency Injection

Tasks receive all dependencies automatically‚Äîno manual wiring required:

  • Connectors: Access as self.input_connector, self.output_connector
  • Services: Access as self.openai_service, self.crm_service
  • Policies: Access as self.consumer_policy

Lifecycle Hooks

Each task type provides hooks for handling outcomes:

Consumer Hooks

  • process_transaction(transaction) ‚Äî Your business logic
  • handle_process_success(transaction, result) ‚Äî Called after success
  • handle_process_exception(transaction, exception) ‚Äî Called on failure

Producer Hooks

  • prepare_transaction(transaction) ‚Äî Prepare for dispatch
  • handle_prepare_success(transaction, result) ‚Äî Called after success
  • handle_prepare_exception(transaction, exception) ‚Äî Called on failure

Concurrency Models

Choose sync or async based on your workload:

  • Synchronous: Thread-based parallelism. Best for CPU-bound work or libraries that don't support async.
  • Asynchronous: Non-blocking concurrency. Best for I/O-bound work with high concurrency requirements.
