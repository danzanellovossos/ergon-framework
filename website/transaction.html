Transaction

The atomic, immutable unit of work at the heart of Ergon. Everything else exists to serve it.

What is a Transaction?

A Transaction is the fundamental unit of work in Ergon. It represents a single, atomic piece of data that must be processed indivisibly. This could be a message from a queue, a record from a file, an event from a stream, or an entire batch that must be treated as one unit.

The key insight is that "atomic" does not mean "small." A transaction is whatever the external system considers a single, cohesive unit. If Kafka delivers a batch of 1,000 records as one message, that entire batch becomes one Transaction.

Transaction Structure

Every Transaction is an immutable model with three fields:

from pydantic import BaseModel, ConfigDict
from typing import Any, Dict

class Transaction(BaseModel):
    id: str                          # Unique identifier for tracing
    payload: Any                     # The actual data content
    metadata: Dict[str, Any] = {}    # Contextual information
    
    model_config = ConfigDict(frozen=True)  # Immutable

üÜî id

A unique identifier assigned when the transaction is created. Used for distributed tracing, logging correlation, and debugging.

üì¶ payload

The actual data. Intentionally typed as Any because different connectors produce different data types.

üìã metadata

Additional context: routing keys, headers, timestamps, file paths, or any information that helps with processing.

Immutability Guarantee

The Transaction model is frozen (frozen=True). Once created, it cannot be modified. This immutability provides critical benefits:

üîí Thread Safety

Immutable objects can be safely shared across threads without locks.

üîç Debugging

You can always trust that the transaction you see is exactly what was received.

üéØ Atomicity

Since transactions cannot be modified, they cannot be partially processed.

Atomicity Rules

Atomicity governs all behavior in the framework:

  1. A transaction must not be split. If the upstream system declares something as a cohesive batch, the framework preserves it intact.
  2. A transaction must not be merged. Two independent messages cannot be arbitrarily grouped into a single transaction.
  3. Transaction boundaries define task semantics. Tasks operate only on transactions‚Äînever on raw transport-specific structures.

Transaction Lifecycle

Every transaction follows a predictable life through the system:

  1. Origin: A service fetches or receives a unit of work from an external system.
  2. Construction: The connector wraps it into an immutable Transaction object.
  3. Ingestion: The task receives the Transaction through a consumer mixin.
  4. Processing: The task applies pure, idempotent business logic to the payload.
  5. Hooks: Success or exception handlers are invoked.
  6. Production: Optionally, new Transactions are created for outbound emission.
  7. Output: The connector extracts payloads and sends them via the service.
  8. Telemetry: Logs, metrics, and spans are correlated using the transaction ID.

Why Transaction-First?

By placing the Transaction at the center of the architecture, Ergon achieves:

  • Consistency: Every component speaks the same language‚Äîtransactions.
  • Testability: Test business logic with plain Transaction objects. No mocks needed.
  • Scalability: Scaling decisions are made at the transaction level.
  • Observability: Every unit of work has a unique ID for end-to-end tracing.
